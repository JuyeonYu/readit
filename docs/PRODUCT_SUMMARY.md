# MessageOpen 프로덕트 요약

> 기획/마케팅 논의용 문서
> 최종 업데이트: 2026-02-04

---

## 1. 서비스 한 줄 소개

**"링크로 메시지를 보내고, 상대가 읽었는지 알 수 있는 서비스"**

---

## 2. 해결하는 문제

| 상황 | 문제 |
|------|------|
| 중요한 제안서를 보냈는데... | 상대가 읽었는지 알 수 없음 |
| 메시지가 스팸으로 갔나? | 확인할 방법이 없음 |
| "확인하셨나요?" 연락하기 | 어색하고 관계에 부담 |

**핵심 Pain Point**: 메시지를 보낸 후의 불확실성

---

## 3. 우리의 해결책

1. **메시지를 링크로 변환** → 어디서든 공유 가능 (카톡, 슬랙, 이메일, 문자 등)
2. **상대가 읽으면 알림** → 이메일로 즉시 알려줌
3. **추가 보안 옵션** → 비밀번호, 만료 시간, 1회 읽기 설정 가능

---

## 4. 사용자 흐름

### 보내는 사람 (송신자)
```
메시지 작성 → 링크 생성 → 원하는 채널로 공유 → 읽음 알림 수신
```

### 받는 사람 (수신자)
```
링크 클릭 → [읽기] 버튼 클릭 → 메시지 확인
```
- 회원가입 불필요
- 앱 설치 불필요

---

## 5. 핵심 기능

| 기능 | 설명 |
|------|------|
| 멀티 채널 공유 | 이메일, 슬랙, 카카오톡, 문자 등 어디든 공유 가능 |
| 실시간 읽음 알림 | 상대가 읽는 즉시 이메일 알림 |
| 비밀번호 보호 | 민감한 내용에 비밀번호 설정 |
| 만료 시간 설정 | 1일/7일/30일 후 자동 만료 |
| 1회 읽기 | 한 번 읽으면 사라지는 메시지 |
| 읽기 통계 | 몇 명이 몇 번 읽었는지 확인 |

## 7. 타겟 사용자

### 1차 타겟
- **프리랜서/1인 사업자**: 제안서, 견적서 전달 후 확인 필요
- **영업/세일즈 담당자**: 팔로업 타이밍 파악 필요
- **채용 담당자**: 오퍼 레터 확인 여부 필요

### 2차 타겟
- 중요한 개인 메시지를 보내는 일반 사용자
- 민감한 정보를 안전하게 전달해야 하는 사람

### 사용자 특징
- 이메일 읽음 확인이 안 되는 환경에서 일함
- 다양한 채널(카톡, 슬랙 등)로 소통
- "읽었는지 확인하는 게 눈치 보임"을 느끼는 사람

---

## 8. 경쟁 서비스와 차별점

| 경쟁 | 우리 |
|------|------|
| 이메일 트래킹 (Mailtrack 등) | 이메일뿐 아니라 **모든 채널**에서 사용 가능 |
| 일회용 메시지 (Privnote 등) | 읽음 알림 + 보안 옵션 **모두 제공** |
| 메신저 읽음 표시 | 메신저 **없이도** 읽음 확인 가능 |

**핵심 차별점**: 채널에 관계없이, 링크 하나로 읽음 확인

---

## 9. 현재 상태 (MVP)

### 완성된 것
- 메시지 작성 및 링크 생성
- 읽음 알림 (이메일)
- 비밀번호 보호, 만료 설정, 1회 읽기
- 회원 가입/로그인 (매직 링크)
- 요금제 (Free/Pro)
- 결제 연동 (Lemon Squeezy)

### 미완성 / 향후 계획
- 팀 요금제
- Slack/Webhook 알림
- 통계 대시보드 고도화
- 링크 템플릿

## 10. 논의 포인트

### 기획 관련
1. **무료 사용자 전환**: 월 2회 제한이 적절한가? 더 늘려야 할까?
2. **1차 타겟 집중**: 프리랜서/영업 외에 다른 세그먼트는?
3. **수신자 경험**: "감시당하는 느낌" 없이 자연스러운 UX란?

### 마케팅 관련
1. **메시지**: "읽음 확인"보다 더 매력적인 표현은?
2. **채널**: 타겟에게 도달할 최적의 채널은? (Product Hunt, 인디해커, 링크드인...)
3. **소셜 프루프**: 초기 사용자 확보 및 리뷰 전략
4. **가격 인식**: $12/월이 적절한가? 앵커링 전략은?
